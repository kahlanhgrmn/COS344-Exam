1.2Valid vector for a 3×3 homogeneous matrix multiplication
Answer
D — [2 1 1]ᵀ (column vector, w = 1)

Key Concept In 2D homogeneous coordinates, a point is [x, y, 1]ᵀ. The last component (w) must be 1 for a finite point. It must be a column vector for matrix-vector multiplication.

  • [2 1] — only 2 elements, can't multiply with a 3×3 matrix ✗
  • [2 1 0]ᵀ — w=0 means a direction vector, not a point; translations won't apply ✗
  • [2 1 1] — row vector, wrong orientation ✗
  • [2 1 1]ᵀ — correct: 3-element column vector with w=1 ✓
1.3Scaling matrix applied to [e, e², 0]ᵀ — what transformation?
Answer
F — A transformation not mentioned in the other options

Trap The matrix looks like a scaling matrix, but the vector has w = 0. In homogeneous coordinates: w=0 means a direction vector (point at infinity), not a 2D point.

Applying the matrix gives [3e, 3e², 0]ᵀ — still w=0. You cannot convert this to Cartesian coordinates (x/0 is undefined). No meaningful geometric transformation of a real point has occurred — hence F.

Question 2 — Colour, Lighting & Reflections
2.1Why is a camera sensor a 2D raster image input?
  • A camera sensor is a physical 2D grid of photosensors (rows × columns).
  • Each photosensor captures light at one discrete location → one pixel value.
  • The output is a finite 2D grid of colour values — the definition of a raster image.
  • It is an input device because it digitises the real world into pixel data.
2.2Name the scale from [0,0,0]ᵀ to [1,1,1]ᵀ in the RGB cube
Answer
Greyscale (achromatic scale)

Every point on the diagonal has equal R=G=B, producing neutral greys: [0,0,0]=black → [0.5,0.5,0.5]=grey → [1,1,1]=white. This line is the greyscale axis of the RGB cube.

2.3Implement determineFinalColour() — alpha compositing
Formula
final = α × foreground + (1 − α) × background
Colour determineFinalColour(Colour foreground, Colour background, float alpha) {
    float r = alpha * foreground.r + (1.0f - alpha) * background.r;
    float g = alpha * foreground.g + (1.0f - alpha) * background.g;
    float b = alpha * foreground.b + (1.0f - alpha) * background.b;
    return Colour(r, g, b);
}

α=1 → fully opaque (shows foreground). α=0 → fully transparent (shows background). Applied per channel independently.

2.4What type of reflection gives reflected direction [−2, 3, −1]ᵀ?
Answer
Specular (mirror) reflection

Verify using the reflection formula r = d − 2(d·n)n:

1
Incident direction (light→surface): d = [2,1,3]−[4,4,4] = [−2,−3,−1]
2
Surface normal: n = [0,1,0]
3
d·n = (−3)(1) = −3
4
r = [−2,−3,−1] − 2(−3)[0,1,0] = [−2,−3,−1] + [0,6,0] = [−2, 3, −1] ✓

The calculated specular reflection exactly matches the given direction → the surface is a specular (mirror-like) reflector obeying angle-of-incidence = angle-of-reflection.

Question 3 — Rendering Theory
3.1"Rendering is the process by which the object is moved around the screen" — True or False?
Answer
FALSE

Motivation: Rendering is the process of converting a 3D scene description into a 2D image. Moving objects is done through geometric transformations (translation, rotation, scaling) — a separate step that happens before rendering.

3.2Two methods for how objects influence pixels
a) Name the two methods and how they differ

1. Object-order rendering (Rasterisation): Loop over each object/triangle → find which pixels it covers → colour those pixels. Fast, GPU-friendly, used in real-time graphics.

2. Image-order rendering (Ray Tracing): Loop over each pixel → cast a ray into the scene → find which object it hits → compute colour. Physically accurate but expensive.

Key difference: Object-order iterates objects first; image-order iterates pixels first.

b) Which method does OpenGL use? Why?
Object-order rendering (Rasterisation)

Reason: OpenGL takes each triangle, projects it to the screen, then fills (rasterises) the pixels it covers. It does not cast rays from pixels. This pipeline is optimised for GPU parallel processing.

3.3Why is ray-triangle intersection critical in ray tracing?

Why triangles?

  • Every 3D model is decomposed into triangles (triangulation/tessellation).
  • Triangles are always planar and always convex — 3 points always define a unique flat plane. Quads can be non-planar.
  • Any polygon can be split into triangles → triangles are universal.
  • GPU hardware is built around triangle processing.

Why it matters for ray tracing: Since all objects are made of triangles, testing ray-triangle intersection means you can test against any 3D scene. If you can intersect one triangle, you can intersect the entire world.

5.3What does gl.enable(gl.GL_BLEND) do?

Enables alpha blending. Without it, new fragments simply overwrite the framebuffer. With blending enabled, OpenGL combines the incoming colour with the existing colour using alpha (transparency) values, allowing transparent/semi-transparent objects to render correctly.

You also need: glBlendFunc(GL_SRC_ALPHA, GL_ONE_MINUS_SRC_ALPHA) to specify the blend equation.

6.2Problems with edge-pixel colouring solutions
a) Blend the colours

Problem: The pixel gets processed and written twice (once per triangle) causing double-blending. The result is brighter or differently coloured than intended — a visible artefact along every shared edge on opaque surfaces.

b) Give the edge pixel no colour

Problem: Every shared edge in the entire mesh will have a gap of uncoloured pixels, producing visible dark cracks/holes along all triangle boundaries — completely unacceptable for solid surfaces.

c) Use an offscreen point

Problem: The pixel is always assigned to exactly one triangle based on the offscreen point's side. If the two triangles have different colours, the edge pixel consistently shows only one colour, which can look visually inconsistent. Also fails at viewport boundaries where the offscreen point may not be valid.

1.4Texture lookup f(x,y,z) = (u,v), where u=2x² mod z, v=log(y) mod z
a) Primary bijectivity issue
C — The x² term causes multiple inputs to map to the same output

A bijective function maps each input to a unique output (one-to-one). The term x² is NOT one-to-one: x=2 and x=−2 both give x²=4, so different 3D coordinates map to the same UV → not bijective.

b) Distortion of size
B — Both u and v components are affected, resulting in overall size distortion

u = 2x² mod z grows quadratically → non-uniform size distortion in u. v = log(y) mod z grows logarithmically → non-uniform distortion in v. Both components are distorted (though differently), causing overall size distortion across the texture.

c) Effect of modulus on texture shape
D — The non-linear modulus operation creates discrete jumps → significant shape distortion

The modulus operator causes the UV value to wrap around abruptly (e.g. 9 mod 5 = 4, then 10 mod 5 = 0 — a sudden jump). These discrete discontinuities break the smooth shape of textures, creating distorted/torn-looking projections.

Question 4 — Culling
4.1Which faces are culled for each technique?
a) View Volume Culling

Any object (or part of an object) that lies outside the camera's view frustum/volume is culled. The entire object is discarded before rasterisation. Objects partially outside are clipped at the volume boundary.

In the scenario: objects positioned beyond the far plane, before the near plane, or outside the left/right/top/bottom bounds are culled.

b) Occlusion Culling

Objects completely hidden behind other opaque objects are culled. The GPU doesn't bother rendering geometry that will never be seen. In the scenario: parts of the floor behind the glass panel might be occluded from the camera's view by other geometry.

c) Backface Culling

Triangle faces whose normal points away from the camera are culled. For a floor: the bottom faces of the floor tiles face downward (away from the camera above) and are culled. For the glass panel: the face pointing away from the camera is culled.

4.2Why is winding order important for backface culling?

The winding order of a triangle's vertices (clockwise or counter-clockwise when viewed from outside) determines which direction its normal points.

  • Counter-clockwise (CCW) winding → normal points outward (toward viewer for a front face)
  • Clockwise (CW) winding → normal points inward (away from viewer)

Backface culling discards any face whose normal points away from the camera. If the winding order is inconsistent, the GPU can't tell front from back — it may cull visible faces (creating holes) or keep invisible faces (wasting GPU time). Consistent winding order is essential for culling to work correctly.

4.3C++ code: cull FRONT faces, clockwise winding order
glEnable(GL_CULL_FACE);         // enable culling
glCullFace(GL_FRONT);           // cull front-facing faces
glFrontFace(GL_CW);             // define clockwise as "front"

glCullFace(GL_FRONT) removes front-facing polygons (normally you'd use GL_BACK). glFrontFace(GL_CW) tells OpenGL that clockwise-wound triangles are the "front" faces.

Question 5 — Texture Mapping
5.1Best geometrically determined coordinate technique for the floor
Answer
Planar projection (orthographic/planar texture mapping)

Motivation: The floor is a flat, rectangular surface. Planar projection maps texture coordinates directly from the x and z world-space coordinates of each vertex (treating the floor as lying in the XZ plane). Because the floor is flat, there is no distortion — the texture maps straight onto the surface without wrapping or stretching artifacts that would appear with spherical or cylindrical mapping.

5.2How does tiling work with geometrically determined coordinates?

By scaling the UV coordinates before sampling (multiplying the texture coordinate by a tiling factor), the UV values exceed [0,1] and wrap around — this is tiling.

Set the texture wrap mode to GL_REPEAT: glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_WRAP_S, GL_REPEAT)

Then scale UVs in the shader: vec2 tiledUV = texCoord * tilingFactor. A tiling factor of 4 repeats the texture 4× along each axis across the floor.

5.3aVertex shader — displace vertex along normal by texture lookup
#version 330 core
layout(location = 0) in vec3 vertexPosition;
layout(location = 1) in vec3 surfaceNormal;
layout(location = 2) in vec2 texCoord;

uniform sampler2D customTexture;
out vec3 fragmentColor;

void main() {
    float displacement = texture(customTexture, texCoord).r;
    vec3 displaced = vertexPosition + surfaceNormal * displacement;
    gl_Position = vec4(displaced, 1.0);
    fragmentColor = vec3(displacement, displacement, displacement);
}
5.3bWhat is the name of this texture mapping technique?
Answer
Displacement Mapping

Reason: The texture value is used to physically move (displace) vertices along their normals — it creates real geometric detail, unlike bump/normal mapping which only fakes shading without moving geometry. The result is actual 3D surface deformation driven by the texture.

Question 7 — LOD, Performance, Meshes
7.1aHow to apply LOD to the glass with tiny scratches

Level of Detail (LOD) uses different mesh/texture resolutions depending on the distance from the camera:

  • Close up: High-detail model with actual scratch geometry (many polygons, detailed normal/bump map).
  • Medium distance: Simplified mesh with scratch detail baked into a normal/bump map only (no geometry).
  • Far away: Low-poly glass panel — scratches are invisible at this distance anyway, so don't render them.

When the camera moves, the engine switches between LOD levels. Since tiny scratches are only visible close up, this saves significant rendering time when the glass is far from the camera.

7.1bWhy develop for computationally weak (mobile) platforms first?

If a render runs well on a weak device, it will definitely run on a stronger one — but not vice versa.

  • Designing for weak hardware forces you to write efficient code: lower polygon counts, smaller textures, fewer draw calls, memory-efficient data structures.
  • These optimisations benefit all platforms — the render will look great and run fast everywhere.
  • A render designed only for powerful hardware may be unoptimised and unrunnable on mobile — porting becomes a major rework rather than a minor adjustment.
7.1cHow do old games run on hardware that didn't exist when they were made?

Games communicate with hardware through abstraction layers — APIs like OpenGL, DirectX, or Vulkan. The game talks to the API, not directly to the GPU.

When new hardware is built, the manufacturer implements drivers that make the new GPU behave like the same API the old game uses. The game doesn't know or care about the specific hardware — only the API contract. So as long as the new hardware's drivers support the same API (or an emulation layer bridges them), the old game runs fine.

7.2Manifold violations — index-based meshes

Strict manifold: Every edge is shared by exactly 2 triangles (closed/watertight), and each vertex's triangles form a single connected fan.
Relaxed manifold: Every edge is shared by at most 2 triangles (open boundaries allowed), but vertex fans must still be connected (no disconnected triangle groups around one vertex).

a) Strict manifold violations
i) (2,4,1), (4,1,0), (4,5,1)
VIOLATES — edge 4-1 is shared by all 3 triangles (max allowed = 2)

Check edge 4-1: appears in (2,4,1), (4,1,0), AND (4,5,1) → 3 triangles share this edge → violates the max-2 rule.

ii) (2,4,1), (4,5,1), (2,5,1), (1,0,3)
VIOLATES — vertex 1 has disconnected triangle fans

No edge is shared by 3+ triangles. But triangle (1,0,3) shares no edge with the other three triangles through vertex 1 → the triangles around vertex 1 form two disconnected groups, violating the connected-fan requirement.

b) Relaxed manifold violations
i) (1,0,3), (1,3,5), (0,1,2)
DOES NOT VIOLATE — no edge >2, all vertex fans connected

Edge 1-0 shared by (1,0,3) and (0,1,2) ✓. Edge 1-3 shared by (1,0,3) and (1,3,5) ✓. All other edges appear once. Around each vertex, the triangles form a connected fan. Open boundary edges are allowed under relaxed rules.

ii) (2,1,4), (4,5,1), (4,2,5), (1,3,0)
VIOLATES — vertex 1 has disconnected triangle fans

No edge is shared by 3+ triangles. But vertex 1 appears in (2,1,4), (4,5,1), and (1,3,0). The first two share edge 1-4 so they're connected, but (1,3,0) shares no edge with them through vertex 1 → disconnected fan around vertex 1 → violates relaxed manifold.

Question 8 — Visualisation
8.1Why is clipping negative in rendering but useful in visualisation?

In regular rendering (negative): If the camera clips through a wall or object, you see the model's hollow interior — missing geometry, broken immersion. It breaks the believability of the scene.

In visualisation (positive): Deliberately cutting through a 3D dataset (medical scan, geological layer, architectural model, airflow simulation) reveals the interior cross-section. This is exactly what the user needs — seeing internal structures that would otherwise be hidden by the outer surface. The "flaw" becomes a feature that aids understanding.

8.2Derived dimensionality vs another dimensionality reduction method
Derived Dimensionality

Adds an extra visual dimension to data by encoding a variable as a visual property (colour, size, shape, transparency). The original data dimensions are kept; a new visual channel represents an additional data attribute. Example: a 2D scatter plot where a 3rd variable is encoded as point colour.

  • Advantage: No data loss — all original information is preserved.
  • Disadvantage: Can become visually cluttered/confusing if too many visual channels are used simultaneously.
  • Use case: Showing population, GDP, and continent on a single world map (x=GDP, y=population, colour=continent).
Dimensional Slicing / Filtering

Shows a subset of the data by fixing some dimensions and only displaying a 2D or 3D slice. Example: showing one time step of a 4D (x,y,z,t) dataset at a time.

  • Advantage: Simple, uncluttered — the viewer focuses on one slice at a time.
  • Disadvantage: The user only sees part of the data at once — context from other slices is lost.
  • Use case: Medical imaging — displaying one axial cross-section of a CT scan at a time.
