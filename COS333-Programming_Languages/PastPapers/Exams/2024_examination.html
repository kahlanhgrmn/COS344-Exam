Q1 1 mark Evaluation Criteria Trade-offs

There are often trade-offs between the programming language evaluation criteria. Select the statement that most correctly describes a likely trade-off when readability improves.

  • A. The cost of execution becomes worse.
  • B. The cost of maintenance becomes worse.
  • C. The cost of implementation becomes worse.
  • D. Reliability becomes worse.
Explanation

When readability improves, programs are easier to understand — but this often requires additional language constructs, syntactic sugar, or verbose syntax that a compiler/interpreter must parse and process. This increases the complexity of the language implementation (the compiler/interpreter itself becomes more complex and expensive to build). Improved readability generally helps reliability (errors are easier to spot) and maintenance (easier to understand and fix). Execution cost is not directly worsened by readability.

Q5 5 marks Scheme — removePositiveOddValues Scheme

Write a Scheme function named removePositiveOddValues, which receives one parameter: a simple list containing only numeric atoms. The function should yield (not print) a list containing the values in the parameter list, in their original order, with all positive odd values removed.

Examples:

(removePositiveOddValues '(3 9))
; => ()   (all values are positive odd)

(removePositiveOddValues '(-5 3 0 5 9))
; => (-5 0)   (3, 5, 9 are positive odd and are removed)

Allowed built-in functions:

  • Function construction: lambda, define
  • Binding: let
  • Arithmetic: +, -, *, /, abs, sqrt, remainder, modulo, min, max
  • Boolean values: #t, #f
  • Equality predicates: =, >, <, >=, <=, even?, odd?, zero?, negative?, eqv?, eq?
  • Logical predicates: and, or, not
  • List predicates: list?, null?
  • Conditionals: if, cond, else
  • Quoting: quote, '
  • List manipulation: list, car, cdr, cons
  • I/O: display, printf, newline, read
Your Answer Scheme
Memo Solution
(define (removePositiveOddValues lst)
  (cond
    ((null? lst) '())
    ((and (> (car lst) 0) (odd? (car lst)))
     (removePositiveOddValues (cdr lst)))
    (else
     (cons (car lst) (removePositiveOddValues (cdr lst))))
  )
)
Explanation

Strategy: Recurse through the list element by element. Base case: empty list → return empty list. For each element: if it is BOTH positive (`> 0`) AND odd (`odd?`), skip it (don't include it) and recurse on the tail. Otherwise, include it (cons it onto the result) and recurse. Example: `(removePositiveOddValues '(-5 3 0 5 9))` → -5 is not positive odd → include; 3 is positive odd → skip; 0 is not positive odd → include; 5 is positive odd → skip; 9 is positive odd → skip. Result: (-5 0). The `and` predicate checks both conditions simultaneously.

Examples:

?- doublePositives([-4, -9], X).

?- doublePositives([-3, 6, 7, -1, -5], X).
% X = [-3, 12, 14, -1, -5]

Hint: Comparison operators (<, =<, >, >=) can be used as terms in Prolog. Multiplication is represented by .*

Requirements:

  • Only use constants, variables, list manipulation, arithmetic, relational expressions, and built-in predicates from the textbook/slides and hints. Do not use if-then, if-then-else, or complex system predicates.
Your Answer Prolog
Memo Solution
doublePositives([], []).
doublePositives([H|T], [D|T2]) :-
  H > 0,

Q7 1 mark Binding Time

Binding can occur at different times. Select the option that most correctly describes the binding time of a standard 32-bit signed integer type in a statically typed language to either a sign-and-magnitude or two's complement representation.

  • A. Language design time
  • B. Language implementation time
  • C. Compile time
  • D. Load time
  • E. Runtime
Explanation

Q8 1 mark C# Scope Rules

Consider the following program code in C# (line numbers for reference):

1. {
2.    int y;
3.    {
4.       int x = 5;
5.    }
6.    int x = 2;
7. }

Select the most correct statement.

  • A. The provided program code does not compile because the variable y is not initialised.
  • B. The provided program code does not compile because the variable x on line 4 hides the variable x on line 6.
  • C. The provided program code does not compile because the variable x on line 6 hides the variable x on line 4.
  • D. The provided program code compiles and runs successfully.
Explanation

In C#, the scope of a local variable is from its declaration to the end of its enclosing block. `x` on line 4 is declared inside the inner block (lines 3–5) — its scope ends at line 5. `x` on line 6 is declared in the OUTER block after the inner block ends. These are in DIFFERENT scopes and don't overlap. In C# (unlike Java), this is perfectly legal because the inner `x` is out of scope by the time the outer `x` is declared. C# does NOT flag this as a hiding error the same way Java does. `y` is declared but not initialised, however C# only requires initialisation before USE — if `y` is never used after declaration, no compile error occurs.

Q9 1 mark Character String Types

Consider the following statements about character string types in programming languages, and select the most correct.

  • A. In programming languages that represent character strings using only classes, character strings are always mutable.
  • B. In programming languages that represent character strings using only classes, character strings are always immutable.
  • C. Limited dynamic length strings have a higher overall hardware resource cost than static length strings, and a lower overall hardware resource cost than dynamic length strings.
  • D. C supports character strings as a primitive type.
Explanation

Static length strings have fixed size — simple and cheap. Limited dynamic length strings can grow up to a maximum size — they require extra memory for tracking current length and maximum length, and some runtime overhead for bounds checking. Fully dynamic length strings (like Python strings or Java StringBuffers) can grow without limit — they require heap allocation, possibly reallocation (when capacity is exceeded), and garbage collection overhead. So the ordering is: static < limited dynamic < fully dynamic in terms of hardware resource cost. Strings represented as classes can be either mutable or immutable — this is a language design choice, not mandatory either way.

Q10 1 mark Unions

Consider the following statements about unions, and select the most correct.

  • A. Unions are closely related to classes in object-oriented programming, but are more limited in terms of functionality.
  • B. Discriminated unions are more memory efficient than free unions.
  • C. Unions can be used as a mechanism for returning multiple values from a subprogram.
  • D. Unions are used when memory cost is of great concern.
Explanation

The primary purpose of a union is to save memory by allowing multiple fields to share the same memory space. Only one field is valid at a time, but all fields overlay the same bytes — the union occupies only the size of its largest member, not the sum of all member sizes. This is useful in memory-constrained environments. Unions are NOT closely related to classes (classes support encapsulation, inheritance, and methods; unions are purely about memory layout). Discriminated unions are LESS memory efficient (they add a tag field). Unions can store one value at a time — they cannot return multiple values simultaneously.

Q13 1 mark Short-Circuit Evaluation

Consider the following program code in a hypothetical programming language:

int count = 5
double result = 0.0

while ((count > 0) and (result = (10.0 / count))) {
   print result
   count = count - 1
}

Assume C-like syntax, assignment as an expression, and for logical AND, and integers are interpreted as Boolean values like in C. Select the most correct statement.

  • A. The provided program code produces an error regardless of whether the logical and is short-circuit evaluated or not.
  • B. The provided program code produces an error if the logical and is short-circuit evaluated.
  • C. The provided program code produces an error if the logical and is not short-circuit evaluated.
  • D. The provided program code produces no error regardless of whether the logical and is short-circuit evaluated or not.
Explanation

The while condition is `(count > 0) and (result = (10.0 / count))`. When `count` reaches 0: if `and` IS short-circuit evaluated, `count > 0` is false → the second operand `10.0/count` is NOT evaluated → no division by zero. If `and` is NOT short-circuit evaluated, BOTH operands are always evaluated → `10.0 / 0` is evaluated when count=0 → division by zero error. The assignment `result = (10.0/count)` is a side effect inside the condition — this is intentional for reading the next value. So the error ONLY occurs when short-circuit evaluation is absent.

Consider the following statements about unary assignment operators, and select the most correct.

Explanation

Unary assignment operators like `++` and `--` in C/C++ can be used as both prefix (`++i`) and postfix (`i++`) — the same symbol means different things depending on position. Q20 1 mark Pass-by-Name

Consider the following program code in a hypothetical programming language:

doSomething(int first, int second) {
   first = 2
   print(first)
   print(" ")
   print(second)
}
void main() {
   int val1 = 5
   int val2 = val1 + 1
   foo(val1, val2)
}

Provide the output of the program if the language uses pass-by-name for all parameter passing. If an error occurs, write Error. (Note: the second print call outputs a single space.)

Answer

2 6

Explanation

Pass-by-name: every occurrence of a formal parameter in the subprogram body is textually replaced by the actual parameter. `doSomething(val1, val2)` → actual parameters are `val1` and `val2`. After substitution: `first = 2` becomes `val1 = 2`; `print(first)` becomes `print(val1)`; `print(second)` becomes `print(val2)`. Execution: val1=5, val2=6 initially. Then val1 = 2 (val1 changes to 2). print(val1) → prints 2. But val2 was originally `val1 + 1` — with pass-by-name, `val2` is the NAME 'val2', and val2 was computed as val1+1 = 5+1 = 6 BEFORE the call. However, with pass-by-name, the second actual parameter is the expression `val2` (a variable name), not `val1 + 1`. So `second` → `val2` (which still = 6 at this point, since val2 = val1+1 was computed at declaration time and val1 was 5 then). print(val2) → 6. Output: 2 6.

Q26 1 mark Subclasses and Subtypes

Imagine a hypothetical programming language in which a subclass can inherit a public method from a parent class and change its access control to private or protected. Select the most correct statement.

  • A. In the hypothetical programming language, dynamic binding is less efficient.
  • B. In the hypothetical programming language, dynamic binding is impossible.
  • C. In the hypothetical programming language, subclasses are subtypes.
  • D. In the hypothetical programming language, subclasses are not subtypes.
Explanation

A subclass is a subtype of its parent if subclass objects can be used anywhere a parent object is expected — i.e., the subclass honours all the same public interface as the parent. If a subclass can CHANGE inherited public methods to private or protected, then a subclass object can no longer fulfil all the promises of the parent class (some public methods disappear from the external view). This violates the Liskov Substitution Principle — subclass objects cannot substitute for parent objects. Therefore, the subclass is NOT a subtype of the parent. Dynamic binding is still possible (it's about method lookup, not type relationships), so it is NOT made impossible.

Select only the approaches that make Smalltalk more orthogonal than Java. (Incorrectly selected options will be penalised.)

Q28 1 mark OOP in C++ vs C#

Consider the following statements about object-oriented programming in C++ and C#, and select the most correct.

  • A. C++ provides worse support for multiple inheritance than C# does.
  • B. C++ provides less writability in terms of nested classes than C# does.
  • C. C++ uses a more orthogonal approach to object-oriented programming than C# does.
  • D. C++ provides a more readable approach to dynamic message binding than C# does.
Explanation

In C++, virtual methods are marked `virtual` at the declaration — any method NOT marked virtual is statically bound. The programmer explicitly opts in to dynamic binding. In C#, to dynamically bind, the parent method must be marked `virtual` AND the overriding method marked `override` (and the overriding method must explicitly use the `new` keyword when hiding). This double-declaration requirement in C# is more verbose and potentially confusing. C++'s single `virtual` keyword is simpler and more readable for indicating dynamic binding. C++ supports multiple inheritance (BETTER than C# in this regard, not worse). C++ nested classes work similarly to C# for basic cases.
