Q1 1 mark SIMULA 67 Design Influences

Various factors can influence programming language design. Consider the following programming language design influences, and select the one that had its earliest impact on the design of SIMULA 67.

  • A. Computer architecture.
  • B. Programmer efficiency becoming more important than machine efficiency.
  • C. The move from process oriented programming to data oriented programming.
  • D. The move to object-oriented programming.
Explanation

SIMULA 67 was created by Dahl and Nygaard to model real-world systems (ships, factories, etc.). The key design influence was the shift from process-oriented thinking (like Fortran) to data-oriented thinking, where entities in the simulation are modelled as data objects with associated operations. This led to the concept of classes and objects. The move to OOP was the RESULT of SIMULA 67, not an influence on it. Computer architecture influenced early languages like Fortran I. Programmer efficiency becoming more important than machine efficiency influenced later languages in the 1970s-80s.

Q2 2 marks Hybrid Implementation System Select all that apply

Consider the following programming language evaluation criteria, and select only those that benefit from the use of a hybrid implementation system. (Incorrectly selected options will be penalised.)

  • A. Readability.
  • B. Writability.
  • C. Cost.
  • D. Portability.
Explanation

A hybrid implementation system compiles source code to an intermediate form (bytecode) which is then interpreted by a virtual machine. (1) Cost: a hybrid system reduces distribution and porting costs because a single bytecode distribution runs on any platform that has the virtual machine installed, eliminating the need to recompile for every target architecture. (2) Portability: programs compiled to an intermediate form can run on any machine with the VM, making them highly portable. Readability and writability are properties of the language syntax, not its implementation method.

Q3 1 mark Plankalkül Syntax

Consider only the syntax used by Plankalkül for an assignment using arrays and subscripts. Consider the following statements, and select the one that is the most correct.

  • A. The syntax had poor readability.
  • B. The syntax had poor writability.
  • C. The syntax had poor reliability.
  • D. The syntax had poor execution cost.
Explanation

Plankalkül (Zuse, 1945) used a two-dimensional, multi-row notation for expressing operations involving arrays and subscripts. The subscript was written below the variable name, and the type was written in yet another row. This multi-level notation made even simple assignments very difficult to read. The syntax was notably of poor readability. While it was also not particularly writable, readability is the most clearly cited weakness in the textbook.

Q4 1 mark ALGOL 60 and I/O

The language specification of ALGOL 60 lacked support for I/O operations. Consider the following statements, and select the one that most correctly describes a consequence this lack of support had on a main design goal of ALGOL 60.

  • A. The reliability of ALGOL 60 suffered.
  • B. The execution cost of ALGOL 60 suffered.
  • C. The execution cost of ALGOL 60 improved.
  • D. The machine independence of ALGOL 60 suffered.
Explanation

ALGOL 60's primary design goal was machine independence (portability). I/O operations are inherently machine-dependent, so the language designers intentionally left I/O out of the specification to maintain machine independence. However, every implementation then added its own non-standard I/O facilities, which meant that any ALGOL 60 program requiring I/O was NOT portable across different implementations. The very goal (machine independence) that motivated omitting I/O ended up suffering because programs needed I/O and got it through incompatible, implementation-specific mechanisms.

Q6 5 marks Scheme — doublePositives Scheme

Write a Scheme function named doublePositives, which receives one parameter: a simple list containing only numeric atoms. The function should yield (not print) a list containing the same values as the parameter list, but with positive values doubled. Negative and zero values remain unchanged.

Examples:

(doublePositives '())
; => ()   (no elements in the parameter list)

(doublePositives '(2 -4 7 -3))
; => (4 -4 14 -3)   (2→4, 7→14; -4 and -3 are unchanged)

> Note: Start your program with #lang racket.

Allowed built-in functions:

  • Function construction: lambda, define
  • Binding: let
  • Arithmetic: +, -, *, /, abs, sqrt, remainder, modulo, min, max
  • Boolean values: #t, #f
  • Equality predicates: =, >, <, >=, <=, even?, odd?, zero?, negative?, eqv?, eq?
  • Logical predicates: and, or, not
  • List predicates: list?, null?
  • Conditionals: if, cond, else
  • Quoting: quote, '
  • List manipulation: list, car, cdr, cons
  • I/O: display, printf, newline, read
Your Answer Scheme
Memo Solution
(define (doublePositives lst)
  (cond
    ((null? lst) '())
    ((> (car lst) 0)
     (cons (* 2 (car lst)) (doublePositives (cdr lst))))
    (else
     (cons (car lst) (doublePositives (cdr lst))))
  )
)
Explanation

Strategy: Recurse through the list. Base case: empty list → return empty list. For each head element: if it is positive (> 0), double it and cons it onto the result of recursing on the tail. Otherwise (zero or negative), keep it unchanged and cons it onto the result. The cond checks for null first, then positivity, then else handles zero and negatives. Example: (doublePositives '(2 -4 7 -3)) → 2 is positive → cons 4; -4 is not positive → cons -4; 7 is positive → cons 14; -3 is not positive → cons -3. Result: (4 -4 14 -3).

Write a Prolog proposition called doubleNonMatching(L1, X, L2), which succeeds if the list L2 contains the same values as the list L1, without their order being modified, but with all values that do not match X doubled. All values that do match X should remain unchanged.

Examples:

?- doubleNonMatching([], 2, X).
% X = []

?- doubleNonMatching([3, 2, 4, 4, 2], 2, X).
% X = [6, 2, 8, 8, 2]

Hint: In Prolog arithmetic, represents multiplication.*

Requirements:

  • You may define additional helper propositions.
  • Only use constants, variables, list manipulation, arithmetic, relational expressions, and built-in predicates from the textbook/slides. Do not use if-then, if-then-else, or complex system predicates.
Your Answer Prolog
Memo Solution
doubleNonMatching([], _, []).
doubleNonMatching([H|T], X, [H|T2]) :-
  H =:= X,
  doubleNonMatching(T, X, T2).
doubleNonMatching([H|T], X, [D|T2]) :-
  H =\= X,
  D is H * 2,
  doubleNonMatching(T, X, T2).
Explanation

Three clauses: (1) Base case: empty list → empty result. (2) If head H equals X (H =:= X for numeric equality): keep H unchanged, prepend it to result, recurse on tail. Q8 1 mark Variable Lifetime Categories

Consider the following program code in a hypothetical programming language:

void main() {
   var myVariable = "Test"
   print(myVariable)

   var anotherVariable = 5
   myVariable = anotherVariable * 2
   print(myVariable)
}

Select the option that most correctly describes the category of myVariable in terms of lifetime.

  • A. Static variable.
  • B. Stack-dynamic variable.
  • C. Explicit heap-dynamic variable.
  • D. Implicit heap-dynamic variable.
Explanation

myVariable is first assigned a string ('Test'), then later reassigned an integer (anotherVariable * 2). The variable changes type at runtime based on assignment, and storage is allocated implicitly by the assignment itself — the programmer does not explicitly allocate (no malloc/new). This is characteristic of an implicit heap-dynamic variable: storage is allocated on the heap automatically when the variable is assigned, and the type binding also occurs at assignment time. Static variables exist for the program lifetime. Stack-dynamic variables are created when their block is entered. Explicit heap-dynamic variables require explicit allocation calls.

Q9 1 mark Named Constants

Consider the following two subprograms in a hypothetical programming language:

void f1() {
   constant LENGTH = 3
   var firstArray[LENGTH] = {6, 8, 7}
}
void f2() {
   constant INPUT_LENGTH
   readInteger(INPUT_LENGTH)
   var secondArray[INPUT_LENGTH]
}

The special word constant is the only way to declare a named constant. Subprogram f1 is legal, while subprogram f2 is illegal and will not compile. Name the type of named constants supported in this programming language.

Answer

Manifest constants

Explanation

f1 declares `constant LENGTH = 3` — the value 3 is known at compile time and is legal. f2 declares `constant INPUT_LENGTH` and then reads a value into it at runtime — the value is NOT known at compile time, which is illegal in this language. A language that only allows named constants whose values are known at compile time (literal values or constant expressions) supports only MANIFEST constants. Manifest constants must have their values specified at compile time, embedded directly in the source code.

Q10 1 mark List Comprehensions in Python

  • B. List comprehensions increase the writability of Python.
  • C. List comprehensions improve the reliability of Python.
  • D. List comprehensions reduce the execution cost of Python.
Explanation

This makes it easier and faster to write list-manipulation code, increasing writability. Readability is debatable — simple comprehensions are readable, but complex nested ones can reduce readability. Reliability is not obviously improved (comprehensions can silently produce empty lists). Execution cost is not reduced (comprehensions have similar cost to explicit loops).

Q11 1 mark Weak Typing in C++

Q12 1 mark Arithmetic Operators in Ruby vs Java

Both Ruby and Java are object-oriented programming languages. Consider the following statements about arithmetic operators in Ruby and Java, and select the one that is the most correct.

  • B. Arithmetic operators in Ruby are always as reliable as arithmetic operators in Java.
  • C. Arithmetic operators in Ruby are implemented in a more orthogonal way than arithmetic operators in Java.
  • D. Arithmetic operators in Ruby have a lower execution cost than arithmetic operators in Java.
Explanation

In Ruby, arithmetic operators (+, -, *, /, etc.) are implemented as methods on objects. Every number is an object, and `2 + 3` is syntactic sugar for `2.+(3)`. Ruby's uniform method-based approach is more orthogonal.

Q13 1 mark Unary * Operator in C++

Consider the following statements about the unary * operator in C++, and select the one that is the most correct.

  • A. The unary * operator is poorly designed, because it is overloaded to produce very different results depending on the context in which it is used.
  • B. The unary * operator is poorly designed, because C++ could have been designed to avoid its use while leaving the language's flexibility and conciseness unchanged.
  • C. The unary * operator is well designed, because it is not overloaded.
  • D. The unary * operator is well designed, because it cannot be confused with any non-unary operators.
Explanation

The * symbol in C++ is used as: (1) the binary multiplication operator (a * b), (2) the unary dereference operator (*p), and (3) in declarations to indicate a pointer type (int *p). These produce very different semantic results — multiplication, memory access, and type declaration respectively. Using the same symbol for such different purposes is a form of overloading that reduces readability and can confuse programmers. The unary * IS effectively overloaded (different meaning in different contexts), so the option claiming it is well-designed because it is not overloaded is incorrect.

Q15 1 mark Mixed-Mode Assignments in Ada vs C

Consider the following statements about mixed-mode assignments in Ada and C, and select the one that is the most correct.

  • A. Mixed-mode assignments in Ada are less readable than mixed-mode assignments in C.
  • B. Mixed-mode assignments in Ada are less writable than mixed-mode assignments in C.
  • C. Mixed-mode assignments in Ada are less reliable than mixed-mode assignments in C.
  • D. Mixed-mode assignments in Ada have a higher execution cost than mixed-mode assignments in C.
Explanation

Ada enforces very strict type rules with almost no implicit type coercions. Q16 1 mark Ruby vs Perl Nested Selector Disambiguation

Q18 1 mark Counter-Controlled Loops in Ada vs C

Consider the following statements about counter-controlled loops in Ada and C, and select the one that is the most correct.

  • B. Counter-controlled loops are less readable in Ada than in C.
  • C. Counter-controlled loops are more reliable in Ada than in C.
  • D. Counter-controlled loops have a higher cost of execution in Ada than in C.
Explanation

Ada's for loop (`for i in 1..10 loop`) implicitly declares the loop variable i, makes it READ-ONLY inside the loop body (it cannot be modified), and uses a fixed range. Q20 1 mark Default Parameter Values

Consider the following statements about default values for subprogram parameters, and select the one that is the most correct.

  • A. There is no way for a default parameter value to appear anywhere but the beginning of a subprogram's list of parameters.
  • B. There is no way for a default parameter value to appear anywhere but the end of a subprogram's list of parameters.
  • C. It is possible for a default parameter to appear anywhere in a subprogram's list of parameters if positional parameters are used.
  • D. It is possible for a default parameter to appear anywhere in a subprogram's list of parameters if keyword parameters are used.
Explanation

With keyword parameters, the caller names each argument explicitly (e.g., `func(second=10)`), so ANY parameter can be omitted regardless of its position — the language knows which argument corresponds to which parameter by name. Q22 1 mark Pass-by-Result Parameters

Consider the following statements about pass-by-result subprogram parameters, and select the one that is the most correct.

  • A. Pass-by-result parameters can only copy a value to the caller.
  • B. Pass-by-result parameters can only transfer an access path to the caller.
  • C. Pass-by-result parameters can either copy a value to the caller, or transfer an access path to the caller.
  • D. Pass-by-result can neither copy a value to the caller, nor transfer an access path to the caller.
Explanation

Pass-by-result parameters transfer a value ONE WAY: from the subprogram back to the caller (at the end of execution). There are two valid implementation strategies: (1) Physical copy — when the subprogram ends, the final value of the formal parameter is copied back into the actual parameter variable. (2) Access path — the actual parameter's address is passed; at the end of the subprogram, the formal parameter's value is written through the access path to the caller's variable. Both implementations achieve the same result. Neither 'only copy' nor 'only access path' is the unique method.

Q23 2 marks Referencing Environments — Ad-hoc and Shallow Binding

Consider the following program code in a hypothetical programming language:

var A = 12
fun sub1() {
   print A
}
fun sub2(param) {
   var A = 35
   call param()
}
fun main() {
   var A = 22
   sub2(sub1)
}

Assume the programming language allows subprograms to be passed as parameters, uses static scoping, and that execution starts with main. Provide the output if the following types of binding are used to determine the referencing environment of the passed subprogram.

Answer

{'ad_hoc_binding': '12', 'shallow_binding': '35'}

Explanation

The language uses static scoping. sub1 is defined at global scope where var A = 12. When sub1 is passed to sub2 and then called via param(): (1) Ad-hoc binding: the referencing environment used is determined by sub1's lexical definition point (global scope). In the global scope, A = 12. So sub1 prints 12. (2) Shallow binding: the referencing environment is the CURRENT DYNAMIC ENVIRONMENT at the point where sub1 is actually called, which is inside sub2. sub2 has its own local A = 35. So sub1 sees sub2's A and prints 35. Note: Deep binding (not asked here) would capture the environment when sub1 is PASSED as a parameter — that would be main's scope where A = 22.

Q27 1 mark Java Generics — Number of Versions

Consider the following program code in Java:

public class MyClass<T> {
   private T data;
   public MyClass(T param) {
      data = param;
   }
   public T getData() {
      return data;
   }
}

Also assume the following main method is provided:

public static void main(String[] args) {
   MyClass<String> a = new MyClass<String>("test");
   Integer myInt = new Integer(12);
   MyClass<Integer> b = new MyClass<Integer>(myInt);
}

Identify the number of versions of MyClass that exist after compile time. Type only the digit.

Answer

1

Explanation

Java generics use TYPE ERASURE. At compile time, the type parameter T is erased and replaced with its upper bound (Object, since no bound is specified for MyClass<T>). The compiler creates exactly ONE version of MyClass — MyClass<Object> effectively — and inserts explicit casts wherever T is used. At runtime, the JVM has no knowledge of the original generic types. This differs from C# (which creates separate versions per type argument at runtime) and C++ (which creates separate versions per type argument at compile time). Java creates 1 version.

Q28 1 marks Multiple Inheritance — Implicit Name Collision Select all that apply

Consider a hypothetical programming language that supports object-oriented programming with multiple inheritance. Four classes A, B, C, and D are defined. Classes B and C both inherit only from class A. Class D inherits from both B and C. An implicit name collision is possible. Select only the implementation details that together result in an implicit name collision (not explicit). (Incorrectly selected options will be penalised.)

  • A. Defining a method named myMethod in class A.
  • B. Defining a method named myMethod in class B.
  • C. Defining a method named myMethod in class C.
  • D. Defining a method named myMethod in class D.
Explanation

An IMPLICIT name collision arises when the same method name reaches D through MULTIPLE INHERITANCE PATHS without being explicitly redefined in B or C. If A defines myMethod, both B and C INHERIT it without modification. D then inherits from both B and C — getting two inherited paths to the same A method — creating a diamond problem / implicit collision. The compiler cannot determine which version to use without explicit disambiguation. If B or C redefined myMethod, the collision would be EXPLICIT (deliberately introduced by B or C). Defining in D itself resolves rather than causes a collision.

Q29 2 marks C++ vs Smalltalk Execution Cost Select all that apply

C++ has a lower cost of execution than Smalltalk. Select only the factors that contribute to the lower cost of execution in C++. (Incorrectly selected options will be penalised.)

  • A. The approach C++ uses in relation to the exclusivity of classes.
  • B. The approach C++ uses to perform message binding.
  • C. The fact that C++ supports heap allocated objects.
  • D. The fact that subclasses are not necessarily subtypes in C++.
Explanation

(1) C++ does NOT require everything to be an object — primitive types (int, float, etc.) are not objects and have no object overhead. Smalltalk treats EVERYTHING as an object, including integers and booleans — every operation involves object method dispatch, adding significant overhead. C++'s non-exclusive use of objects reduces cost. (2) C++ uses STATIC binding by default — only methods explicitly declared `virtual` use dynamic dispatch. Smalltalk uses DYNAMIC binding for ALL message sends, always consulting a method lookup table at runtime. C++'s predominantly static binding is far more efficient. Heap allocation and subtype relationships do not significantly differentiate the two in terms of overall execution cost.
