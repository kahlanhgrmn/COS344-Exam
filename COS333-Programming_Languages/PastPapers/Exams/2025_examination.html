Q1 1 mark Cost of Programming Languages (Modern Context)

Consider the following types of cost that can be associated with a programming language, and select the one that is most relevant to consider in a modern computing context.

  • A. Cost of compiling programs
  • B. Cost of executing programs
  • C. Cost of reliability
  • D. Cost of the language implementation system
Explanation

In modern computing, hardware is cheap and fast, compilers are free and optimised, and language implementation systems are widely available at no cost. The dominant cost is now reliability: unreliable software causes financial losses, security breaches, regulatory penalties, and reputational damage. Critical systems (banking, healthcare, infrastructure) must be correct. Poor reliability leads to expensive emergency patches and maintenance. Therefore, the cost of reliability is the most important factor in a modern context.

Q2 1 mark Portability

Consider the following programming languages, and select the one that has the highest level of portability.

  • A. Fortran I
  • B. PL/I
  • C. Swift
  • D. Java
Explanation

Java compiles source code to platform-independent bytecode that runs on any machine with a JVM (Java Virtual Machine) installed. This 'write once, run anywhere' model gives Java the highest portability of the options. Fortran I compiles to machine-specific code for the IBM 704 — essentially zero portability. PL/I is compiled to platform-specific binaries. Swift is primarily for Apple platforms. Java's bytecode/JVM approach is specifically designed for cross-platform portability.

Q3 1 mark Readability

Consider the following programming languages, and select the one that had the highest overall readability.

  • A. C++
  • B. BASIC
  • C. Ada
  • D. C#
Explanation

Ada was designed by the US Department of Defense with readability and maintainability as explicit primary goals — it was meant for long-lived, safety-critical systems where code must be read and maintained by many people over decades. Ada has verbose, self-documenting syntax (explicit type declarations, named parameter passing, strong typing). C++ is a complex hybrid language with many syntactic shortcuts that reduce readability. BASIC was designed for simplicity but not for large-scale readable programs. C# is modern and readable but was designed after readability was well-understood — Ada is the most explicitly readability-focused design.

Q4 1 mark Well-Definedness

Consider the following programming languages, and select the one that focused the most on the well-definedness of the original language design.

  • A. Fortran I
  • B. ALGOL 68
  • C. C
  • D. C++
Explanation

ALGOL 68 was designed with extreme rigour using a formal two-level van Wijngaarden grammar to precisely and completely define the language. Every construct was formally specified with no ambiguity or undefined behaviour. Fortran I had many implementation-dependent behaviours. C is infamous for extensive undefined behaviour (buffer overflows, pointer misuse, etc.). C++ inherits all of C's undefined behaviours and adds more. ALGOL 68 is the standout example of a language designed with well-definedness as a central concern.

Q5 5 marks Scheme — getZerosDoublePositives Scheme

Write a Scheme function named getZerosDoublePositives, which is applied to one parameter: a simple numeric list lst. The function should yield (not print) a list containing the values in lst, in their original order, where zero values remain unchanged, positive values are doubled, and negative values are omitted.

Examples:

(getZerosDoublePositives '())
; => ()

(getZerosDoublePositives '(-5))
; => ()   (negative values are omitted)

(getZerosDoublePositives '(-1 4 0 -3 12))
; => (8 0 24)   (-1 and -3 omitted; 4→8, 12→24; 0 unchanged)

Allowed built-in functions:

  • Function construction: lambda, define
  • Binding: let (complex forms like let* and named let are not allowed)
  • Arithmetic: +, -, *, /, abs, sqrt, remainder, modulo, min, max
  • Boolean values: #t, #f
  • Equality predicates: =, >, <, >=, <=, even?, odd?, zero?, negative?, eqv?, eq?
  • Logical predicates: and, or, not
  • List predicates: list?, null?
  • Conditionals: if, cond, else
  • Quoting: quote, '
  • List manipulation: list, car, cdr, cons
  • I/O: display, printf, newline, read
Your Answer Scheme
Memo Solution
(define (getZerosDoublePositives lst)
  (cond
    ((null? lst) '())
    ((negative? (car lst))
     (getZerosDoublePositives (cdr lst)))
    ((zero? (car lst))
     (cons (car lst) (getZerosDoublePositives (cdr lst))))
    (else
     (cons (* 2 (car lst)) (getZerosDoublePositives (cdr lst))))
  )
)
Explanation

Strategy: Recurse through the list. Base case: empty list → return empty list. For each element: (1) If negative (`negative?`) → skip (don't include), recurse on tail. (2) If zero (`zero?`) → include unchanged, recurse on tail. (3) Otherwise (positive) → double it (`* 2`) and include, recurse on tail. Example trace for `(-1 4 0 -3 12)`: -1 is negative → skip; 4 is positive → 8; 0 is zero → 0; -3 is negative → skip; 12 is positive → 24. Result: (8 0 24). The `cond` checks in order, so negative is checked before zero before positive.

Q6 5 marks Prolog — getNegativesNegatePositives Prolog

Write a Prolog proposition called getNegativesNegatePositives(L1, L2), where L1 and L2 are both simple numeric lists. The proposition succeeds if L2 contains the values in L1 in their original order, where negative values remain unchanged, positive values are negated, and zero values are omitted.

Examples:

?- getNegativesNegatePositives([0], X).
% X = []   (zero values are omitted)

?- getNegativesNegatePositives([1, 0, -2, 0], X).
% X = [-1, -2]   (1 negated to -1; -2 unchanged; zeros omitted)

Requirements:

  • You may define additional helper propositions.
  • Only use constants, variables, list manipulation, arithmetic, relational expressions, the is operator, cuts, and the built-in not proposition. Do not use if-then, if-then-else, the semicolon (;) or pipe (|) operator outside of a list, or complex system predicates (including built-in member, append, reverse).
Your Answer Prolog
Memo Solution
getNegativesNegatePositives([], []).
getNegativesNegatePositives([H|T], L2) :-
  H =:= 0,
  getNegativesNegatePositives(T, L2).
getNegativesNegatePositives([H|T], [N|L2]) :-
  H > 0,
  N is -H,
  getNegativesNegatePositives(T, L2).
getNegativesNegatePositives([H|T], [H|L2]) :-
  H < 0,
  getNegativesNegatePositives(T, L2).
Explanation

Four clauses: (1) Base case: empty list → empty result. (2) If H is zero: skip it (don't include in L2), recurse on tail. (3) If H is positive (H > 0): negate it (N is -H) and prepend N to L2, recurse. (4) If H is negative (H < 0): keep it unchanged, prepend H to L2, recurse. The `is` operator performs arithmetic. Example: `getNegativesNegatePositives([1, 0, -2, 0], X)` → 1 positive → -1; 0 → skip; -2 negative → -2; 0 → skip. X=[-1,-2].

Q7 1 mark Special Words

Consider the following program code in a hypothetical programming language:

main() {
   var switch
   write("Do you wish to continue?")
   read(switch)

   while(switch) {
      var input
      write("Which action would you like to perform?")
      read(input)
      switch (input) {
         case 1: call action1()
                 break
         case 2: call action2()
                 break
      }
      write("Do you wish to continue?")
      read(switch)
   }
}

Identify and fill in the kind of special word that switch is in this program.

Answer

Keyword

Explanation

In the program, `switch` is used on line `var switch` as a variable name AND as a construct `switch (input) { ... }`. For this to be legal in the same program, the language must allow `switch` to be used as an identifier in non-switch-statement contexts. This is the definition of a KEYWORD: a word that is only special in specific syntactic contexts and CAN be redefined as a variable name elsewhere. A RESERVED WORD cannot be used as an identifier at all — if `switch` were reserved, line `var switch = ...` would cause a compile error.

Q8 1 mark Global Variables — PHP vs Python vs C

Consider the following statements about accessing and modifying global variables in PHP, Python, and C, and select the most correct.

  • A. PHP's approach to accessing and modifying global variables is less writable than both Python and C.
  • B. PHP's approach to accessing and modifying global variables is less reliable than both Python and C.
  • C. PHP's approach to accessing and modifying global variables is less readable than both Python and C.
  • D. PHP's approach to accessing and modifying global variables has a lower cost of execution than both Python and C.
Explanation

In PHP, EVERY function that accesses or modifies a global variable must declare it with `global $varName;` — even just to read it. In Python, you can READ a global variable freely; you only need `global varName` to MODIFY it. In C, global variables are accessible directly without any declaration inside functions — complete transparency. PHP's requirement to explicitly declare globals in every function for every access adds noise to the code and clutters readability. C's transparent globals and Python's read-transparent globals are more readable than PHP's always-required declarations.

Q9 1 mark Elemental Array Operations in Fortran

Fortran supports elemental array operations, which are not supported in most other high-level languages. Select the reason that describes a strong justification for their inclusion in Fortran.

Fortran was designed for scientific and engineering computation, where array and matrix operations are fundamental. Q10 1 mark List Comprehensions in Python

Consider the following statements about list comprehensions in Python, and select the most correct.

Q11 1 mark Operator Precedence and Associativity in APL

Consider the following statements about operator precedence and associativity in APL, and select the most correct.

  • A. Operator precedence and associativity in APL is highly orthogonal.
  • B. Operator precedence and associativity in APL is very complex due to the large number of operators.
  • C. Operator precedence and associativity in APL introduces a high cost of execution.
  • D. Operator precedence and associativity in APL is unreliable.
Explanation

APL has NO operator precedence hierarchy — all operators have equal precedence and associate from right to left. This is a single, uniform rule applied consistently to ALL operators, regardless of which operator it is. One rule, no exceptions, no special cases. This uniformity is the essence of orthogonality: a small set of principles (one precedence rule, one associativity rule) applied to everything. While this can produce counterintuitive results compared to mathematical conventions, it is undeniably a highly orthogonal design.

Q12 1 mark Operators in Ruby

Consider the following statements about operators in Ruby, and select the most correct.

  • A. Operators in Ruby are implemented in a way that is not very orthogonal.
  • B. Operators in Ruby are not very writable.
  • C. Operators in Ruby have a high cost of execution.
  • D. Operators in Ruby are very reliable.
Explanation

In Ruby, arithmetic and other operators are implemented as METHODS — `a + b` is actually the method call `a.+(b)`. Every operator use involves a method dispatch: looking up the method in the object's class hierarchy, checking access, and calling the method. This is significantly more expensive than a direct hardware arithmetic instruction. Ruby is known for being slower than compiled languages partly because of this method-dispatch overhead on every operation. Ruby operators ARE orthogonal (they're consistent methods), ARE writable, and ARE reliable within the dynamic type system.

Q13 1 mark Mixed Mode Arithmetic — F# vs C++

Consider the following statements about mixed mode arithmetic expressions in F# and C++, and select the most correct.

  • A. Mixed mode arithmetic expressions in F# are less readable than in C++.
  • B. Mixed mode arithmetic expressions in F# are more writable than in C++.
  • C. Mixed mode arithmetic expressions in F# have a higher execution cost than in C++.
  • D. Mixed mode arithmetic expressions in F# are more reliable than in C++.
Explanation

F# (a functional language) does NOT allow implicit mixed-mode arithmetic — you cannot add an int and a float without an explicit type conversion. Every type mismatch must be explicitly resolved by the programmer. C++ allows implicit numeric coercions (int → float, etc.) which silently change types without programmer knowledge. F#'s strict approach means type errors in arithmetic are caught at COMPILE TIME — you can never accidentally mix types. This makes F# arithmetic much more reliable than C++'s implicit coercion approach, which can produce subtle, hard-to-find bugs from unexpected type conversions.

Q14 1 mark Short-Circuit Evaluation

Consider the following program code in a hypothetical programming language:

main() {
   int input
   int total = 0

   while ((total <= 100) or ((input = read()) != EOF)) {
      total = total + input
   }
   write(total)
}

The program reads a file of numbers, adding values to a total until either the total exceeds 100 or the end of file is reached. Assume C-like syntax, assignment is an expression, or is a logical or, and EOF indicates end of file. Select the statement that most correctly describes a problem with this program.

  • A. If the or operator is short-circuit evaluated a side effect will occur, which causes a problem.
  • B. If the or operator is short-circuit evaluated a side effect will not occur, which causes a problem.
  • C. If the or operator is not short-circuit evaluated a side effect will occur, which causes a problem.
  • D. If the or operator is not short-circuit evaluated a side effect will not occur, which causes a problem.
Explanation

The while condition: `(total <= 100) or ((input = read()) != EOF)`. The second operand contains a side effect: `read()` assigns a value to `input`. The program depends on `read()` being called each iteration to get the next number. If `or` IS short-circuit evaluated: when `total <= 100` is TRUE, the second operand is SKIPPED → `read()` is NOT called → `input` is never updated → the program loops forever reading no data (the side effect does NOT occur → problem). If `or` is NOT short-circuit evaluated: both operands always evaluated → `read()` always called → program works correctly.

Q15 1 mark Conditional Targets

Consider the following statements about conditional targets, and select the most correct.

  • A. Conditional targets reduce the writability of assignments.
  • B. Conditional targets increase the readability of assignments.
  • C. Conditional targets increase the cost of execution of assignments.
  • D. Conditional targets reduce the reliability of assignments.
Explanation

A conditional target is when the target (left-hand side) of an assignment is a conditional expression, like `(flag ? x : y) = value` in Perl/C++. This makes it harder for a reader (or the programmer) to know which variable is actually being assigned — you must evaluate the condition to know the assignment target. This reduces reliability because it becomes easy to make mistakes about which variable changes. It does NOT reduce writability (it makes certain patterns more concise). It does NOT increase readability — on the contrary, the target is now harder to identify at a glance.

Q17 1 mark Disambiguating Nested Selectors — Python vs Perl

Q18 1 mark Multiple-Way Selection — Writability

Select the programming language that allows you to achieve this in the most writable way.

  • A. C++
  • B. JavaScript
  • C. C#
  • D. Ruby
Explanation

Q19 1 mark Counter-Controlled Loops in C++

Q20 1 mark User-Located Loop Control — Java vs C++

Q22 1 mark Default Parameters and Keyword Parameters

Some programming languages allow default values to be given to a formal parameter when an actual parameter is left out. Select the option that most correctly describes a mechanism that allows any actual parameter to be left out of a subprogram call, regardless of the order in which the formal parameters are listed.

Q24 1 mark Overloaded Subprograms in C++

The following C++ program code is not legal:

int doSomething(int val) {
   return val + 2;
}

float doSomething(int val) {
   return val + 1.0;
}

Select the option that most correctly describes the underlying reason for this being illegal.

  • A. The fact that C++ does not support overloaded subprograms.
  • B. The fact that variables of different types cannot be added.
  • C. The fact that C++ performs a large number of coercions.
  • D. The fact that C++ performs no coercions.
Explanation

In C++, overloaded functions are disambiguated by PARAMETER PROFILE only (not return type). The two `doSomething` functions have IDENTICAL parameter profiles `(int val)`. The reason return type cannot be used for disambiguation in C++ is that coercions are so prevalent — an int return could be coerced to float or vice versa, making return-type-based disambiguation unreliable and ambiguous. Because C++ performs many coercions, the compiler cannot safely use return type to choose between overloads. Ada, which has far fewer coercions, CAN use return type to disambiguate. The many coercions in C++ are the root cause of this limitation.

Q25 1 mark Closures in JavaScript

Consider the following JavaScript program code:

function doSomething(a, b) {
   return function(y) {return (a + b) - y;}
}

var thing = doSomething(10, 20);

Select the option that most correctly describes what thing is.

  • A. The function function(y) {return (a + b) - y;}
  • B. The function function(y) {return (10 + 20) - y;}
  • C. A reference to the doSomething function.
  • D. An invalid function, because a value for y cannot be specified.
Explanation

Calling `doSomething(10, 20)` returns an anonymous function (a closure) that captures `a=10` and `b=20` from the enclosing scope. The returned function is: `function(y) {return (a + b) - y;}` with a=10 and b=20 permanently bound. So `thing` is the function `function(y) {return (10 + 20) - y;}`. When `thing(5)` is called later, it returns 30-5=25. `thing` is NOT a reference to `doSomething` itself, and it IS a valid function — `y` is the parameter to be supplied when `thing` is called.

Q26 1 mark Pass-by-Result Parameters

Consider the following statements about pass-by-result parameters, and select the most correct.

  • A. Pass-by-result parameters can be passed using a physical copy or an access path.
  • B. Pass-by-result parameters can be passed using only a physical copy because using an access path would create a functional side effect.
  • C. Pass-by-result parameters can be passed using only a physical copy because the formal parameter is deallocated once the function terminates.
  • D. Pass-by-result parameters can be passed using only an access path because a physical copy would take up additional memory and processing time.
  • E. Pass-by-result parameters can be passed using only an access path because a physical copy would cause the calling function to work with an out-of-date copy of the formal parameter.
Explanation

Pass-by-result (out mode) transmits a value FROM the formal parameter BACK TO the actual parameter when the subprogram returns. Two valid implementations: (1) Physical copy — the formal parameter's final value is copied to the actual parameter's memory location on return. This is the standard implementation. (2) Access path — store the address of the actual parameter at call time; on return, write the formal parameter's value through that stored address. Both work correctly for out-mode semantics. An access path does NOT create a functional side effect (changes only happen on return). Neither is strictly 'the only way' — both are valid.

Q27 1 mark Multidimensional Arrays as Parameters — C vs C#

Consider the following statements about multidimensional arrays as subprogram parameters in C and C#, and select the most correct.

  • A. The approach to passing multidimensional arrays in C is more writable than the approach used in C#.
  • B. The approach to passing multidimensional arrays in C is more readable than the approach used in C#.
  • C. The approach to passing multidimensional arrays in C is more reliable than the approach used in C#.
  • D. The approach to passing multidimensional arrays in C is less orthogonal than the approach used in C#.
Explanation

In C, passing a multidimensional array requires specifying all dimensions EXCEPT the first in the formal parameter: `void fun(int matrix[][10])`. This means you need DIFFERENT formal parameters for arrays of different sizes — a 5-column matrix and a 10-column matrix need different parameter declarations. Two separate mechanisms depending on array size is not orthogonal. In C#, arrays are objects with a `.Length` property — you pass them as plain array references with no size specification. One mechanism works for ALL array sizes → more orthogonal. The C approach is less writable (less flexible) and less orthogonal.

Q28 2 marks Implicit Name Collision in Multiple Inheritance Select all that apply

Consider a hypothetical programming language that supports multiple inheritance. Four classes A, B, C, and D are defined. Classes B and C both inherit only from A. Class D inherits from both B and C. An implicit name collision is possible. Select only the implementation details that together result in an implicit name collision (not explicit). (Incorrectly selected options will be penalised.)

  • A. Defining a method named myMethod in class A.
  • B. Defining a method named myMethod in class B.
  • C. Defining a method named myMethod in class C.
  • D. Defining a method named myMethod in class D.
Explanation

An IMPLICIT name collision arises when the collision is a consequence of the inheritance structure itself, not because a programmer explicitly defined the same method in multiple places. In diamond inheritance (D inherits B and C, which both inherit from A): if ONLY A defines `myMethod`, then B inherits it from A and C inherits it from A. Now D inherits from both B and C — and gets TWO copies of A's `myMethod` through two paths. Nobody explicitly redefined it — the collision appeared implicitly from the diamond structure. Defining `myMethod` in B, C, or D would be explicit (deliberate) definitions, not implicit collisions.

Q29 2 marks Implementing Linked List Nodes Select all that apply

You wish to implement a linked list and its nodes. Select only the programming language constructs that can be used to implement the linked list nodes. (Incorrectly selected options will be penalised.)

  • A. A struct in C#
  • B. A struct in C++
  • C. A class in C++
  • D. A union in C++
Explanation

A linked list node needs data field(s) and a self-referential pointer to the next node of the same type. (1) C++ `struct`: structs in C++ are almost identical to classes (only default access differs); they CAN have pointer members of the same struct type: `struct Node { int data; Node* next; };` — valid for linked list nodes. (2) C++ `class`: can have pointer members of the same class type — standard approach for linked lists. (3) C# `struct`: the slides explicitly state C# structs have 'No members referencing the struct itself' — you CANNOT have a `Node* next` field in a C# struct. (4) C++ `union`: shares memory among all members — having both data and next pointer would overlay them, destroying the data when next is accessed.

Q30 1 mark Ruby

Consider the following statements about Ruby, and select the most correct.

  • A. Ruby has highly efficient primitive types.
  • B. Ruby has support for a programmer to explicitly allocate and deallocate heap memory for objects.
  • C. Ruby allows the programmer to choose between dynamic and static message binding.
  • D. Ruby classes have very unreliable message interfaces.
Explanation

In Ruby, class definitions are EXECUTABLE — you can reopen any class at any time and add, remove, or redefine methods. This means the set of methods an object responds to can change during program execution. A method that exists at one point may not exist later (or vice versa). This makes it difficult to guarantee which messages an object will respond to, making the message interface unreliable. Ruby does NOT have efficient primitive types (everything is an object). Ruby does NOT support explicit heap deallocation (uses GC). Ruby uses ONLY dynamic message binding — the programmer cannot choose static binding.

Q31 1 mark Java vs C++ OOP

Consider the following statements about Java and C++, and select the most correct.

  • A. Java uses objects less exclusively than C++.
  • B. Java supports some of the benefits of multiple inheritance, while preventing name collisions.
  • C. Java supports some of the benefits of multiple inheritance, but does not prevent name collisions.
  • D. Java exclusively supports dynamic message binding, while C++ allows the programmer to choose between dynamic and static message binding.
Explanation

Java does not support multiple class inheritance, but it supports implementing MULTIPLE INTERFACES. Interfaces provide benefits of multiple inheritance: a class can take on multiple types, and clients can treat objects as any of those types. Before Java 8, interfaces contained only method DECLARATIONS and named constants — no state (instance variables). This means there is no competing state from multiple 'parents', which is the main source of name collisions in multiple inheritance. Java thus gets many benefits of multiple inheritance while avoiding name collision problems. Java uses objects MORE exclusively than C++ (not less). Java does NOT exclusively use dynamic binding (final/private/static methods use static binding).
