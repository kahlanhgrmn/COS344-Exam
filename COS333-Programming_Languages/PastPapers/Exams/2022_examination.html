Q1 1 mark Systems Programming Language Design

For programming languages that are designed for systems programming applications, the primary concern is efficiency. Consider the following statements, and select the one that most correctly identifies the second most important concern in programming languages designed for systems programming.

  • A. Portability.
  • B. Generality.
  • C. Orthogonality.
  • D. Reliability.
Explanation

For systems programming languages (operating systems, device drivers, embedded systems), the primary concern is efficiency — these programs must run as fast as possible with minimal overhead. The second most important concern is RELIABILITY. Systems software (OS kernels, drivers) must function correctly at all times; a crash or incorrect behavior can bring down an entire system. Portability is actually a low priority for systems code (often deliberately hardware-specific). Generality and orthogonality are secondary concerns not traditionally cited as second-most-important for systems programming.

Q2 1 mark Pseudocode Language Interpretation

Pseudocode languages were interpreted rather than compiled. Consider the following statements, and select the option that most correctly identifies why these languages were interpreted.

  • A. Interpretation offers the advantage of cross-platform compatibility.
  • B. Interpretation is more reliable in terms of error detection than compilation.
  • C. Certain operations were not supported in hardware at the time, meaning execution was slow regardless of whether interpretation was used.
  • D. Interpretation requires less computing resources than compilation does, meaning it worked better on the limited hardware of the time.
Explanation

Early pseudocode languages (1950s) were interpreted rather than compiled. The key reason: many operations (particularly floating-point arithmetic) were NOT supported directly in hardware on early computers. Programs needed software routines to perform these operations, which were inherently slow regardless of whether the program was compiled or interpreted. Since execution was slow either way (due to hardware limitations), the overhead of interpretation was negligible relative to the software-emulated operations. Interpretation was therefore chosen for its simplicity and flexibility. Cross-platform compatibility was not a concern in that era.

Q3 1 mark Objective-C

Consider the following statements, and select the option that most correctly describes Objective-C.

  • B. Object-oriented programming in Objective-C is based on C++.
  • C. Object-oriented programming in Objective-C is based on Smalltalk.
  • D. Objective-C does not support object-oriented programming.
Explanation

Objective-C was developed by Brad Cox in the early 1980s by adding object-oriented programming capabilities to C. The OOP model chosen was based on SMALLTALK's message-passing paradigm, not C++ style. Objective-C uses dynamic message dispatch, objects identified by class, and the syntax for sending messages (`[object message]`) is derived from Smalltalk's approach. Q4 5 marks Scheme — getPositiveOddValues Scheme

Write a Scheme function named getPositiveOddValues, which receives one parameter: a simple list containing only numeric atoms. The function should yield (not print) a list containing only the positive (non-zero) odd values contained in the parameter list.

Examples:

(getPositiveOddValues '())
; => ()   (no elements in the parameter list)

(getPositiveOddValues '(1 2 7 -3 4))
; => (1 7)   (1 and 7 are the only positive odd values; 2 and 4 are positive even; -3 is negative)

Allowed built-in functions:

  • Function construction: lambda, define
  • Binding: let
  • Arithmetic: +, -, *, /, abs, sqrt, remainder, modulo, min, max
  • Boolean values: #t, #f
  • Equality predicates: =, >, <, >=, <=, even?, odd?, zero?, negative?, eqv?, eq?
  • Logical predicates: and, or, not
  • List predicates: list?, null?
  • Conditionals: if, cond, else
  • Quoting: quote, '
  • List manipulation: list, car, cdr, cons
  • I/O: display, printf, newline, read
Your Answer Scheme
Memo Solution
(define (getPositiveOddValues lst)
  (cond
    ((null? lst) '())
    ((and (> (car lst) 0) (odd? (car lst)))
     (cons (car lst) (getPositiveOddValues (cdr lst))))
    (else
     (getPositiveOddValues (cdr lst)))
  )
)
Explanation

Strategy: Recurse through the list element by element. Base case: empty list → return empty list. For each head element: if it is BOTH positive (> 0) AND odd (odd?), include it by consing it onto the result of recursing on the tail. Otherwise (zero, negative, or even), skip it and recurse on the tail without including it. The and predicate checks both conditions. Example: (getPositiveOddValues '(1 2 7 -3 4)) → 1 is positive AND odd → include; 2 is even → skip; 7 is positive AND odd → include; -3 is negative → skip; 4 is even → skip. Result: (1 7).

Examples:

?- stripOccurrences(a, [b, c, d], X).
% X = [b, c, d]

?- stripOccurrences(a, [a, b, a, c, a], X).
% X = [b, c]

Requirements:

  • You may define additional helper propositions.
  • Only use constants, variables, list manipulation, arithmetic, relational expressions, and built-in predicates from the textbook/slides. Do not use if-then, if-then-else, or complex system predicates.
Your Answer Prolog
Memo Solution
stripOccurrences(_, [], []).
stripOccurrences(X, [X|T], L2) :-
  stripOccurrences(X, T, L2).
stripOccurrences(X, [H|T], [H|L2]) :-
  H \= X,

Three clauses: (1) Base case: empty list → empty result (regardless of X). (2) If the head of L1 unifies with X (i.e., equals X): skip it and recurse on the tail, passing the same L2 accumulator. Q6 1 mark Scope and Lifetime of Static Variables

Consider the following statements about the scope and lifetime of static variables, and select the one that is most correct.

  • A. The scope of the variable is from the point of the variable declaration until the end of its scope, while the lifetime is from the start of program execution until the end of program execution.
  • B. The scope of the variable is from the start of program execution until the end of program execution, while the lifetime is from the point of the variable declaration until the end of its scope.
  • C. The scope and lifetime are both from the point of the variable declaration until the end of its scope.
  • D. The scope and lifetime are both from the start of program execution until the end of program execution.
Explanation

Static variables have a SPLIT between scope and lifetime. SCOPE (visibility) is limited: the variable is visible only from its declaration point to the end of the enclosing block/function (e.g., a static local variable in C is only accessible within the function). LIFETIME is unlimited: the variable is allocated when the program starts and persists until the program terminates — it retains its value between function calls. This distinguishes static variables from stack-dynamic variables (whose lifetime matches their scope) and heap-dynamic variables.

Q7 1 mark Implicitly Dereferenced Pointers

It is possible for pointers to be implicitly dereferenced by a programming language. Consider the following possible disadvantages associated with implicitly dereferenced pointers, and select the one that is most correct.

  • A. Void pointers cannot be used.
  • B. Pointer arithmetic is disallowed.
  • C. Pointers cannot be used for indirect addressing.
  • D. Pointers cannot be used for dynamic memory management.
Explanation

If a programming language implicitly dereferences pointers (writing `p` gives the value pointed to, not the pointer itself), then there is no way to operate on the POINTER VALUE directly. Pointer arithmetic requires accessing the pointer's address value (e.g., advancing by sizeof(T) bytes to move to the next element). If `p` always gives the dereferenced value, you cannot compute `p + 1` as 'advance the pointer by one element' — the addition would apply to the pointed-to value instead. Therefore, implicit dereferencing effectively disallows meaningful pointer arithmetic. Void pointers, indirect addressing, and dynamic memory management can still be supported through other mechanisms.

Q8 1 mark Operator Precedence and Associativity in APL

Consider the following statements about operator precedence and associativity in APL, and select the one that is most correct.

  • A. Precedence rules in APL are more orthogonal than precedence rules in other programming languages, but associativity rules in APL are less orthogonal than associativity rules in other programming languages.
  • B. Associativity rules in APL are more orthogonal than associativity rules in other programming languages, but precedence rules in APL are less orthogonal than precedence rules in other programming languages.
  • C. Precedence and associativity rules in APL are both more orthogonal than precedence and associativity rules in other programming languages.
  • D. Precedence and associativity rules in APL are both less orthogonal than precedence and associativity rules in other programming languages.
Explanation

APL uses a single, completely uniform rule for all operators: ALL operators have the SAME precedence level (no hierarchy) and ALL operators associate RIGHT to LEFT. This is maximally orthogonal — one consistent rule applies to every operator without exceptions or special cases. Most other languages have multiple precedence levels (e.g., * before +) and mixed associativity rules (left-to-right for most, right-to-left for assignment), creating many special cases. APL's single rule is more orthogonal (though arguably less intuitive for mathematical expressions).

Q10 1 mark Ada and Assignment Coercions

Ada supports no assignment coercions. Consider the following statements, and select one that most correctly describes a result of this lack of assignment coercions.

  • A. Ada is more writable.
  • B. Ada is more orthogonal.
  • C. Ada is more reliable.
  • D. Ada has lower cost.
Explanation

Ada was designed with minimal implicit type coercions — all type conversions must be EXPLICIT in source code. This means the compiler catches type mismatches at compile time rather than silently coercing values. For example, assigning a float to an integer variable requires an explicit `Integer(floatVar)` conversion call. This prevents accidental precision loss, overflow, or unexpected value changes from silent coercions. The result is a MORE RELIABLE language because type errors are detected early. Ada is actually LESS writable (more verbose explicit conversions) and not more orthogonal due to this restriction.

Q12 1 mark Short-Circuit Evaluation and Side Effects

Consider the following program code fragment in a hypothetical programming language:

int [] myArr = {3, 4, 1, 6, 2}
int index = 0
int length = 5
int threshold = 10
while ((myArr[index] == threshold) or (++index < length))
{ ... }

Select the option that most correctly describes a problem with the above program code.

  • A. If the or operator is short-circuit evaluated, a side effect will not occur, which causes a problem.
  • B. If the or operator is short-circuit evaluated, a side effect occurs, which causes a problem.
  • C. If the or operator is not short-circuit evaluated, a side effect will not occur, which causes a problem.
  • D. If the or operator is not short-circuit evaluated, a side effect occurs, which causes a problem.
Explanation

The condition is: `(myArr[index] == threshold) or (++index < length)`. The `++index` in the second operand is a side effect (it increments index). If `or` IS short-circuit evaluated: when `myArr[index] == threshold` is TRUE, the second operand is NOT evaluated, so `++index` does NOT execute. Index stays at the same position → the loop re-evaluates the same element repeatedly → if the condition remains true, the loop never advances → INFINITE LOOP. The side effect failing to occur causes a problem. If not short-circuit: `++index` always executes, advancing through the array normally.

Q13 2 marks Fortran Three-Way Selection Statement Select all that apply

Fortran I had a three-way selection statement. In this statement, one branch would be executed if the control variable value was above zero, a second branch would be executed if the control variable value was equal to zero, and a third branch would be executed if the control variable value was below zero. Select only the statements that are properties of the three-way selection statement. (Incorrectly selected options will be penalised.)

  • A. The three-way selection statement allows flow control to be written, which cannot be written using two-way selection and pretest logical loops.
  • B. The three-way selection statement is very efficient.
  • C. The three-way selection statement is an example of programmer time being considered more important than execution time.
  • D. The three-way selection statement is too closely based on the IBM 704 hardware.
Explanation

(1) The Fortran I arithmetic IF (`IF (expr) label1, label2, label3`) compiled directly to a single IBM 704 conditional branch instruction (CAS — Compare and Select), which simultaneously evaluated the sign and branched to one of three targets. This was very efficient — a single machine instruction replaced what would otherwise require multiple comparisons and jumps. (2) The IBM 704 had specific hardware support for three-way branching based on sign (negative/zero/positive), and the arithmetic IF was designed to map directly to this instruction — making it hardware-specific and not portable. The control cannot express flow that two-way selection cannot (option A is false). Programmer time was NOT more important than execution time in Fortran I design (option C is false — machine efficiency was paramount).

Q14 1 mark Counter-Controlled Loops in C vs Ada

Counter-controlled loops are supported in C and Ada. Consider the following statements, and select the one that is most correct.

  • A. Counter-controlled loops in C are less writable than counter-controlled loops in Ada.
  • B. Counter-controlled loops in C are more costly than counter-controlled loops in Ada.
  • C. Counter-controlled loops in C are less reliable than counter-controlled loops in Ada.
  • D. Counter-controlled loops in C are more orthogonal than counter-controlled loops in Ada.
Explanation

C's for loop (`for (i=0; i<n; i++)`) allows: modifying the loop variable inside the body, any expression in each control field, break/continue, and non-standard step values. This flexibility enables subtle bugs (accidentally skipping iterations, off-by-one errors, infinite loops from modified counters). Ada's for loop (`for i in 1..10 loop`) implicitly declares the loop variable (READ-ONLY — cannot be modified in the body), uses a fixed range, and has clear semantics. This makes Ada's loop more reliable. C's loop is more writable (more expressive and concise) but less reliable.

Q15 1 mark Conditional Break in a Loop

var a = 11

Answer

Runtime error

Q17 1 mark Python Executable Subprogram Definitions

  • B. Readability is reduced.
  • C. Cost is reduced.

In Python, function definitions (`def`) are executable statements that create function objects at runtime. Q18 1 mark Default Parameters Anywhere in Parameter List

Q19 1 mark Procedures and Side Effects

Procedures are a specific kind of subprogram. Consider the following statements, and select the one that is most correct.

  • B. Procedures are guaranteed to always have side effects.

Explanation

Procedures are subprograms that do NOT return a value through a return statement. Their primary purpose is to PRODUCE EFFECTS — modifying parameters, global variables, files, or other state.     call sub()

    print A
}

9

Q24 3 marks Java Parameterised Classes (Generics)

Briefly outline how the compiler handles parameterised classes in Java 5.0.

Answer

Java 5.0 uses type erasure. The compiler replaces all type parameters with their upper bounds (typically Object), inserts explicit casts at usage sites, and produces a single compiled .class file. Generic type information is not retained at runtime.

Explanation

Java 5.0 generics use TYPE ERASURE for implementation: (1) All type parameters (T, E, K, V, etc.) are replaced by their upper bounds — if no bound is specified, they become Object; if bounded (e.g., T extends Comparable), they become the bound type. (2) The compiler inserts explicit casts wherever a typed value is extracted from a generic structure. (3) Only ONE compiled version of the class is created (unlike C++ templates which create a separate version per type argument). (4) At runtime, the JVM has no knowledge of the original generic type parameters — they are completely erased. This maintains backwards compatibility with pre-Java 5.0 code (which expected Object-based collections), but means runtime type checks on generic parameters are impossible.

Q26 1 mark Multiple Inheritance Implicit Problem

Assume a programming language that supports multiple inheritance. There is a parent class called A. Two classes, called B and C, both inherit directly from A. Finally, a class called D inherits from both B and C. Consider the following statements, and select the one that causes an implicit problem with the described inheritance hierarchy.

  • A. If any class member is defined in class A.
  • B. If any class member is defined in class B.
  • C. If any class member is defined in class C.
  • D. If any class member with the same name is defined in classes B and C.
Explanation

In the diamond pattern (D inherits B and C, both inherit A), if A defines a member (method or field), both B and C INHERIT it without modification. D then has TWO inherited paths to A's member — through B and through C. The compiler cannot determine which version D should use, creating an IMPLICIT collision (the collision arises from the inheritance structure, not from explicit redefinitions in B or C). If B or C explicitly redefined the member (different versions), the collision would be explicit. The solution in C++ without virtual inheritance is to have the member exist twice in D.

Q28 1 mark C++ vs Java Message Binding

C++ and Java both support object-oriented message binding. Consider the following statements, and select the one that is most correct.

  • A. C++ support for message binding is more writable than Java's support.
  • B. C++ support for message binding is more readable than Java's support.

  • D. C++ support for message binding is more reliable than Java's support.

When reading C++ code, the presence or absence of `virtual` immediately tells the programmer how binding works. Java makes ALL non-static, non-final methods dynamically bound by default with no explicit marker — you cannot tell from a single method declaration whether it will be dynamically dispatched. C++'s explicit `virtual` keyword makes binding behaviour more readable. Java's approach is more writable (less to type) but less readable. C++ is also more efficient (not less).

Q29 1 mark Classes vs Structs in C#

Consider the following statements about classes and structs in C#, and select the one that is most correct.

  • A. C# only supports classes, and not structs.
  • B. Classes are more writable than structs in C#.
  • C. Structs are more writable than classes in C#.
  • D. Classes and structs are equally writable in C#.
Explanation

In C#, the syntactic structure for defining and using classes and structs is nearly identical — both use fields, properties, methods, and constructors with the same declaration syntax, differing only in the keyword (`class` vs `struct`). For simple data types, the writability experience is essentially the same. The key differences are semantic (value vs reference type, no inheritance for structs, stack vs heap allocation) rather than syntactic. C# only supports classes is false — structs ARE supported. 'Classes more writable' would only be relevant for polymorphic/inheritance scenarios; 'structs more writable' might apply for simple value types, but generally the syntax parity makes them equally writable.
